# 가상 메모리 학습 프로젝트 (Virtual Memory Study)

이 프로젝트는 운영체제의 가상 메모리(Virtual Memory) 시스템이 어떻게 동작하는지 실제 코드를 통해 확인하고 분석하기 위한 학습 기록입니다.

## 0. 왜 C 언어로 학습하는가?
- **직접적인 주소 접근**: 포인터(`&`, `*`)를 통해 가상 주소를 직접 확인하고 조작할 수 있는 유일한 도구입니다.
- **추상화 제거**: Java나 Node.js처럼 가상 머신(JVM, V8)이나 가비지 컬렉터(GC)라는 "중간 관리자"가 없어서, 운영체제가 프로세스에게 부여하는 순수한 메모리 레이아웃을 관찰하기에 최적입니다.
- **메모리 고정성**: GC가 데이터 위치를 마음대로 옮기지 않으므로 `vmmap` 등의 도구로 분석한 데이터가 신뢰성을 가집니다.

## 1. 학습한 핵심 개념

### 1.1 가상 주소와 물리 주소
- **가상 주소 (Virtual Address)**: 프로세스가 참조하는 논리적인 주소입니다. 모든 프로세스는 자신만의 독립적인 가상 주소 공간을 가집니다.
- **물리 주소 (Physical Address)**: 실제 RAM에 데이터가 저장되는 물리적인 위치입니다. 운영체제의 페이지 테이블(Page Table)이 가상 주소를 물리 주소로 매핑해 줍니다.

### 1.2 ASLR (Address Space Layout Randomization)
- 보안을 위해 프로그램이 실행될 때마다 메모리 영역(Stack, Heap, Data 등)의 시작 주소를 무작위로 배치하는 기술입니다.
- **실험 인사이트**: ASLR이 없다면 모든 프로세스는 동일한 전역 변수 주소(링커 결정)와 동일한 지역 변수 주소(OS의 스택 시작점 결정)를 가지게 됩니다. ASLR이 이 두 가지를 모두 무작위화하므로 실험 결과에서 서로 다른 주소가 관찰된 것입니다.

### 1.3 프로세스 격리 (Process Isolation)
- 각 프로세스는 서로 다른 가상 주소 공간을 사용하므로, 한 프로세스가 다른 프로세스의 메모리를 침범하거나 엿볼 수 없습니다.
- **Docker 비유 (Bridge vs Host)**:
  - **Host 모드**: 호스트의 자원을 직접 공유하므로 격리가 없음. 가상 메모리가 없는 리얼 모드와 유사.
  - **Bridge 모드**: 각 컨테이너가 독립된 사설 IP(가상 주소)를 가지고, 중간에 번역층(MMU/Kernel)을 거쳐 공인 IP(물리 주소)로 연결됨. 이 **주소 번역(Translation)** 과정이 가상 메모리의 핵심이기에 Bridge 모드와 유사함.

### 1.4 Demand Paging (지연 할당)
- 운영체제는 메모리를 미리 할당하지 않고, 프로세스가 실제로 해당 주소에 접근할 때 비로소 물리 메모리(RAM)를 매핑하는 효율적인 방식을 사용합니다.

**Page Fault 상세 흐름**
```
memset()이 가상 주소에 쓰기 시도
    ↓
MMU(Memory Management Unit)가 페이지 테이블 조회 → 매핑 없음
    ↓
Page Fault 발생 → OS가 미리 등록해둔 핸들러로 CPU 자동 점프
    ↓
OS가 빈 물리 RAM 페이지를 찾아 할당
OS가 페이지 테이블에 [가상 주소 → 물리 주소] 매핑 등록
    ↓
CPU가 원래 명령 재실행 → 성공
```
- 10MB `memset()` 시 페이지(4KB) 단위로 약 2,560번의 Page Fault가 순차 발생하며 RAM이 채워짐

### 1.5 CPU와 OS의 관계

**CPU는 "명령 실행기"**
- CPU는 RAM에서 명령어를 하나씩 fetch해서 실행할 뿐, 그 자체는 단순한 실행 장치
- OS도 결국 RAM에 올라간 코드(프로그램)이며, CPU 입장에선 일반 프로그램과 동일한 명령어 덩어리

**권한 분리 (CPU 모드)**
| 모드 | 대상 | 권한 |
| :--- | :--- | :--- |
| **커널 모드 (Kernel Mode)** | OS 커널 | 하드웨어 직접 제어 가능 |
| **유저 모드 (User Mode)** | 일반 프로그램 | 하드웨어 직접 접근 불가, OS 통해야 함 |

- `malloc()` 같은 함수도 내부적으로 OS에 요청하는 **시스템 콜(System Call)**

**인터럽트 & 예외 처리 테이블**
- CPU 내부에는 특수 이벤트 발생 시 "어디로 점프할지" 등록하는 테이블이 존재
- OS 부팅 시 이 테이블을 자신의 핸들러 주소로 채워둠
- Page Fault 발생 → CPU가 자동으로 OS 핸들러로 점프 (OS가 CPU를 "관리"하는 방식)

```
[네 프로그램]  → 유저 모드 실행
      ↓ 시스템 콜 (하드웨어 접근 필요 시)
[OS 커널]      → 커널 모드 실행, 하드웨어 제어
      ↓
[CPU / RAM / 디스크 등 하드웨어]
```

### 1.6 malloc() / memset() 코드 레벨 동작 원리

```c
char *ptr = (char *)malloc(10 * 1024 * 1024);
// → OS에 가상 주소 공간 예약 요청
// → ptr에 시작 가상 주소 저장 (예: 0x1000)
// → 이 시점엔 RAM 할당 없음

memset(ptr, 'A', size);
// → ptr[0] = 'A' (0x1000), ptr[1] = 'A' (0x1001) ... 1바이트씩
// → 첫 접근 시 Page Fault → OS가 RAM 페이지 할당 후 매핑
// → 4KB 페이지 단위로 Page Fault 반복 발생
```

- `char *`로 캐스팅하는 이유: 1바이트 단위 포인터 연산을 위해 (`int *`면 4바이트씩 점프)
- 가상 주소 공간은 **프로세스마다 독립** → 서로 다른 프로세스가 같은 가상 주소를 가져도 실제 물리 주소는 다름

### 1.7 Stack Overflow 핵심 개념

**스택이 여러 개인 이유 = 스레드(Thread)**
- 프로세스 안에서 동시에 여러 작업을 실행하려면 스레드가 필요
- 스레드는 독립된 실행 흐름 → 각자 자신만의 스택이 필요
- 힙, Data, Text는 같은 프로세스의 스레드끼리 **공유**, 스택은 **각 스레드마다 독립**

```
프로세스 (가상 주소 공간 1개)
├── 스레드 1 → 스택 1 (8MB)
├── 스레드 2 → 스택 2 (8MB)
└── 스레드 3 → 스택 3 (8MB)
```

**스택에만 한도가 있는 이유**
- **버그 조기 감지**: 무한 재귀 버그 발생 시 시스템 전체 메모리를 고갈시키기 전에 해당 프로세스만 종료
- **멀티스레드 안정성**: 스택 크기가 고정돼야 가상 주소 공간에 각 스레드 스택을 미리 배치 가능. 가변이면 스레드 간 침범 위험
- **힙은 제한 없는 이유**: `malloc()` / `free()`로 명시적으로 관리하는 영역이라 OS가 굳이 제한할 필요 없음

**Guard Page**
- 스택 한도 바로 아래에 OS가 "접근 금지" 페이지를 배치
- 스택이 한도를 넘는 순간 Guard Page 침범 → OS가 즉시 Segfault로 프로세스 종료
- 힙까지 닿을 기회 자체가 없음

```
높은 주소
┌─────────────┐
│    Stack    │ ← 스택 프레임이 아래로 쌓임
├─────────────┤ ← 스택 한도 (8MB)
│ Guard Page  │ ← 접근 시 즉시 Segfault 💥
├─────────────┤
│     Heap    │
└─────────────┘
낮은 주소
```

**Stack Overflow는 가상 메모리에서 터진다**
- 터진 원인: 가상 주소 공간의 Guard Page 침범
- RAM 용량과 무관 → RAM이 128GB여도 똑같이 7,689번에서 터짐
- 스택 프레임들은 실제로 RAM에 올라가 있었지만(Demand Paging), Guard Page는 매핑 자체가 없어서 접근 불가

| 문제 | 원인 | 해결책 |
| :--- | :--- | :--- |
| **Stack Overflow** | 가상 주소 스택 한도 초과 | 재귀 → 반복문, `ulimit -s` 조정 |
| **힙 메모리 부족** | 물리 RAM 고갈 | RAM 업그레이드, 메모리 최적화 |

### 1.8 운영체제의 메모리 할당 판단 기준
운영체제는 각 영역의 특성에 따라 할당 크기를 다르게 결정합니다.
- **Text & Data (고정 영역)**: 
  - **결정 시점**: 컴파일(빌드) 시점.
  - **판단 근거**: 실행 파일(바이너리)의 헤더 정보. 컴파일러가 전체 코드 양과 전역 변수의 크기를 미리 합산하여 기록해 둠.
  - **특징**: 프로그램 시작 시 즉시 해당 크기만큼 공간을 확보함.
- **Stack**: 운영체제의 기본 설정값(보통 8MB)을 최대치로 부여. 이를 초과하면 Stack Overflow 발생.
  - **제한 이유 1(안전)**: 무한 재귀와 같은 버그 발생 시 시스템 전체 메모리를 고갈시키는 것을 방지.
  - **제한 이유 2(멀티스레드)**: 여러 스레드가 동시에 실행될 때 각자 독립된 실행 흐름을 보장하기 위함.
  - **독립성**: 각 스레드는 공유된 하나의 스택을 나눠 쓰는 것이 아니라, **각자 독립된 스택 공간과 크기 제한**을 부여받음. 64비트의 광활한 가상 주소 공간 덕분에 수많은 스레드에게 독립된 스택 지도를 그려주는 것이 가능함.
- **Heap**: 초기에는 최소한으로 유지하다가, 프로그램의 요청(`malloc`)에 따라 동적으로 확장.
- **전체 공간**: 64비트 시스템에서는 약 128TB에 달하는 거대한 가상 주소 지도를 일단 프로세스에게 부여하고, 실제 사용되는 부분만 RAM에 매핑함.

## 2. 메모리 세그먼트의 구조

가상 메모리 공간은 용도에 따라 여러 세그먼트로 나뉩니다.

### 2.1 영역별 특징 요약
| 영역 | 저장 데이터 | 특징 |
| :--- | :--- | :--- |
| **Text (Code)** | 기계어 코드 | 읽기 전용(Read-only), 프로그램 본체 |
| **Data** | 전역/정적 변수 | 프로그램 시작 시 할당, 종료 시 해제 |
| **Heap** | 동적 할당 데이터 | `malloc()` 등으로 할당, 사용자가 직접 관리 |
| **Stack** | 지역 변수, 함수 정보 | 함수 호출 시 생성, 종료 시 자동 해제 (LIFO) |

### 2.2 Stack vs Heap 집중 비교
- **자동 vs 수동**: Stack은 함수 종료 시 자동으로 청소되지만, Heap은 프로그래머가 직접 `free()`를 호출해야 함.
- **속도 vs 유연성**: Stack은 매우 빠르지만 크기가 제한적이고 수명이 짧음. Heap은 상대적으로 느리지만 큰 데이터를 다룰 수 있고 함수가 끝나도 데이터를 유지할 수 있음.
- **확장 방향**: Stack은 높은 주소에서 낮은 주소로, Heap은 낮은 주소에서 높은 주소로 서로 마주 보며 확장됨.

## 5. vmmap 리포트 분석 가이드

`vmmap [PID]` 출력 결과를 읽는 핵심 방법입니다.

### 4.1 주요 영역 식별
- **`__TEXT`**: 기계어 코드가 위치한 곳. (`r-x` 권한)
- **`__DATA`**: 전역 변수가 위치한 곳. (`rw-` 권한)
  - 실험 데이터의 `global_var`가 이 영역의 `START` 지점임을 확인.
- **`Stack`**: 지역 변수가 위치한 곳.
  - 실험 데이터의 `local_var`가 이 영역의 `END` 주소 근처에 있음을 확인 (스택이 위에서 아래로 자라기 때문).
- **`MALLOC_...`**: 힙(Heap) 영역. `malloc()` 호출 시 이 범위 내 주소가 할당됨.

### 4.2 컬럼 의미
- **VSIZE**: 프로세스에게 할당된 가상 메모리 전체 크기 (지도상의 크기).
- **RESIDENT (실제 RAM 점유)**: 현재 물리 RAM에 실제로 들어있는 총 데이터 크기. **(이 값이 실제 RAM 사용량입니다.)**
- **DIRTY (수정됨)**: **디스크의 실행 파일(Binary)**과 내용이 달라졌거나, 파일에는 없는 데이터가 RAM에만 존재하는 크기.
  - **Clean (절대 불변)**: `__TEXT`(코드), `__DATA_CONST`(상수) 등 파일과 내용이 100% 일치하는 영역.
  - **Dirty (변화의 증거)**: `Stack`, `Heap`, `__DATA`처럼 프로그램 실행 중 값이 변하거나 새로 생성된 모든 영역. **"수정이 가능한 모든 곳은 결국 Dirty가 된다"**고 이해할 수 있음.

### 4.3 MALLOC 구역 분석 (TINY, SMALL, LARGE)
- **TINY/SMALL**: 프로그램 실행 시 시스템 라이브러리(`libc` 등)가 내부적으로 사용하는 기본 메모리 공간. 사용자가 명시적으로 `malloc`을 호출하기 전에도 이미 일정한 `RESIDENT/DIRTY` 값이 존재함.
- **LARGE**: 대용량 메모리 요청 시 할당되는 영역. `VSIZE`만 먼저 잡히고 실제 접근 전까지 `DIRTY`가 0으로 유지되는 **Demand Paging**의 특징을 가장 명확히 보여줌.
- **Guard Page**: 메모리 오버플로우 공격을 막기 위해 영역 사이에 배치된 "접근 금지" 구역.

## 5. Stack Overflow 실험 (2026-03-17)

`stack_overflow_test.c`를 통해 무한 재귀로 스택 한도를 초과시킨 실험입니다.

### 실험 전 vmmap 확인 (PID 21357)
| 영역 | 주소 범위 | VSIZE | RESIDENT |
| :--- | :--- | :--- | :--- |
| **STACK GUARD** | `166fec000-16a7f0000` | 56.0M | 0K |
| **Stack** | `16a7f0000-16afec000` | 8176K | 32K |

- 스택 한도: `0x16a7f0000` (하한선)
- 재귀 시작 전 RESIDENT 32K = `main()` 프레임만 올라간 상태

### 실험 결과
| 항목 | 값 |
| :--- | :--- |
| **Segfault 발생 depth** | 7689번째 호출 시 |
| **마지막 출력된 depth** | 7688 (`0x16a7f0698`) |
| **스택 프레임 1개 크기** | 1,088 bytes (`buffer[1024]` + 함수 오버헤드) |
| **스택 하한선까지 남은 공간** | 0x698 = 1,688 bytes |

### 핵심 인사이트
- depth 7688은 프레임 생성 및 출력 성공. 이후 7689 프레임(1,088 bytes) 생성 시도 시 남은 공간(600 bytes) 부족 → Guard Page 침범 → OS가 Segmentation fault로 즉시 종료
- `printf` 호출 전에 죽었기 때문에 depth 7689는 출력되지 않음
- **스택 프레임 1개(1,088 bytes) × 7,689회 ≈ 8MB** → 계산과 실측이 거의 일치

## 6. mmap() 실험 (2026-03-18)

`mmap_test.c`를 통해 파일을 가상 주소에 매핑하고 Demand Paging을 확인한 실험입니다.

### 실험 결과 (PID 22076)
| 단계 | VSIZE | RESIDENT | DIRTY | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| **1단계: mmap() 직후** | 10.0M | **0K** | **0K** | 가상 주소만 매핑, RAM 미할당 |
| **2단계: 파일 전체 읽기 후** | 10.0M | **10.0M** | **0K** | RAM에 올라옴, 수정 없으니 Clean |

### 핵심 인사이트

**힙 실험과 비교**
| | malloc/memset | mmap |
| :--- | :--- | :--- |
| RESIDENT (접근 전) | 0K | 0K |
| RESIDENT (접근 후) | 10.0M | 10.0M |
| DIRTY (접근 후) | **10.0M** | **0K** |

- Demand Paging 동작은 동일하지만 **DIRTY가 다른 이유**: 힙은 디스크 원본이 없는 새 데이터 → Dirty. mmap은 디스크 파일이 원본이고 읽기만 했으니 RAM = 디스크 원본 → Clean

**Dirty의 정확한 의미**
> "디스크 원본과 RAM의 내용이 달라졌거나, 디스크에 원본이 아예 없는 경우"

| 경우 | Dirty 여부 |
| :--- | :--- |
| Text (코드) | Clean - 실행 중 코드 수정 없음 |
| Data (전역 변수) | Dirty - 실행 중 값 변경 시 원본과 달라짐 |
| Stack / Heap | Dirty - 디스크에 원본 없음 |
| mmap 읽기만 | Clean - 디스크 원본과 일치 |
| mmap 수정 시 | Dirty - 디스크 원본과 달라짐 |

**mmap은 4개 영역(Text/Data/Stack/Heap)과 별개**
- vmmap에서 `mapped file`이라는 독립된 영역으로 표시됨
- 가상 주소 공간을 사용하지만 디스크 파일과 연결된 별도 공간

## 7. fork() + CoW 실험 (2026-03-18)

`fork_cow_test.c`를 통해 프로세스 복제 시 Copy-on-Write 동작을 확인한 실험입니다.

### 실험 결과 (부모 PID 1704 / 자식 PID 1746)
| 단계 | MALLOC_LARGE RESIDENT | MALLOC_LARGE DIRTY | 비고 |
| :--- | :--- | :--- | :--- |
| **1단계: fork() 전** | 10.0M | 10.0M | 부모가 memset('A') 완료 |
| **2단계: fork() 후, 자식 쓰기 전** | **16K** | **16K** | 물리 RAM 복사 안 함 (CoW 공유 중) |
| **3단계: 자식 memset('B') 후** | 10.0M | 10.0M | CoW 발동 → 자식이 RAM 페이지 복사 |
| **4단계: 자식 종료 후 부모 확인** | - | - | 부모 data[0] = 'A' (격리 확인) |

### 핵심 인사이트

**CoW 동작 흐름**
```
부모 memset('A') → RESIDENT 10MB

fork()
└── 자식 VSIZE 10MB (가상 주소 복제)
    자식 RESIDENT 16K (물리 RAM 복사 안 함 = 부모 페이지 공유 중)

자식 memset('B') → CoW 발동
└── OS가 16KB 페이지씩 하나씩 복사
    자식 RESIDENT 10MB, DIRTY 10MB (이제 자기 RAM 소유)
    부모 data[0] = 'A' 그대로 (프로세스 격리)
```

**스레드 공유 vs CoW 공유 비교**
| | 스레드 공유 | fork() + CoW |
| :--- | :--- | :--- |
| 가상 주소 공간 | 1개 (같이 씀) | 각자 독립 |
| 물리 RAM (수정 전) | 공유 | 공유 |
| 물리 RAM (수정 후) | 공유 (그대로) | 각자 독립 (복사됨) |
| 한 쪽이 수정하면? | 다른 쪽도 바뀜 | 다른 쪽은 변화 없음 |
| 목적 | 협력 (의도적) | RAM 절약 (최적화) |

**fork()의 반환값**
- 부모: 자식 PID 반환 → `else` 블록 실행
- 자식: 0 반환 → `if (pid == 0)` 블록 실행
- 같은 코드인데 반환값이 달라서 서로 다른 분기를 탐

**프로세스 종료 코드**
- `exit(0)` / `return 0` → OS에게 "성공" 전달
- `exit(1)` / `return 1` → OS에게 "실패" 전달
- 프로세스 간 신호 역할 (쉘 스크립트, CI/CD 등에서 활용)

## 8. TLB Miss 실험 (2026-03-19)

`tlb_test.c`를 통해 TLB Miss가 성능에 미치는 영향을 측정한 실험입니다.

### CPU 메모리 접근 계층

```
CPU가 가상 주소 접근
        ↓
① TLB 확인 (주소 변환 캐시)
    Hit  → 물리 주소 즉시 획득
    Miss → 페이지 테이블 조회 → 물리 주소 획득 → TLB에 저장
        ↓
② L1 캐시 확인 (데이터 캐시, 128KB)
    Hit  → 데이터 즉시 반환
    Miss → L2 캐시 확인
        ↓
③ L2 캐시 확인 (16MB, Apple Silicon)
    Hit  → 데이터 반환 → L1에도 저장
    Miss → RAM에서 가져옴 → L1/L2에 저장
```

| | 저장하는 것 | Miss시 대안 |
| :--- | :--- | :--- |
| **TLB** | 가상→물리 주소 변환 결과 | 페이지 테이블 조회 (RAM) |
| **L1 캐시** | 실제 데이터 | L2 캐시 확인 |
| **L2 캐시** | 실제 데이터 | RAM 접근 |

### 실험 결과
| 세트 | 페이지 수 | 2번째 패스 | 비고 |
| :--- | :--- | :--- | :--- |
| **작은 세트 (8MB)** | 512 | 6 µs | TLB Hit |
| **큰 세트 (256MB)** | 16,384 | 179 µs | TLB Miss |

### 핵심 인사이트

**Apple Silicon에서 차이가 작은 이유**
- L2 캐시(16MB)가 페이지 테이블 자체를 캐시하고 있음 → TLB Miss 페널티 감소
- 실제 접근 데이터(16KB)가 L1 캐시(128KB)에 다 들어감 → 2번째 패스가 빠름
- Apple Silicon은 TLB + 캐시 계층이 매우 잘 설계되어 있어 순수 TLB Miss 효과를 분리하기 어려움
- x86 서버에서는 동일 실험 시 5~10배 차이가 나는 경우도 있음

**속도 비교**
```
TLB Hit + L1 Hit  → 가장 빠름 (수 ns)
TLB Hit + L2 Hit  → 빠름
TLB Hit + RAM     → 느림
TLB Miss + RAM    → 가장 느림 (수백 ns)
```
